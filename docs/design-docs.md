# Design Document: Feedback Hub

## 1. Overview

The Feedback Hub is a platform for capturing, voting on, and prioritizing user ideas. The goal is to provide a transparent roadmap managed by administrators and fueled by organic user feedback.

## 2. Architectural Decisions and Trade-offs

### 2.1. Backend: Express.js + Clean Architecture

- **Decision:** Use vanilla Express.js instead of highly opinionated frameworks (e.g., NestJS) while implementing Clean Architecture principles.
- **Trade-off:** Higher initial setup time (manual routing, dependency injection) in exchange for total decoupling of business rules. If the web framework changes tomorrow, the Use Cases remain intact.

### 2.2. Database: PostgreSQL + Prisma ORM

- **Decision:** Relational modeling to ensure data integrity (e.g., composite primary key on the votes table to prevent duplication under concurrency). Prisma is used for static typing and development speed for the MVP.
- **Trade-off:** ORMs can hide inefficient queries (like the N+1 problem).
- **Mitigation:** Monitor queries generated by the Prisma Client. If needed for critical listing endpoints, fallback to raw queries or materialized views in the future.

### 2.3. Authentication: JWT via HttpOnly Cookies

- **Decision:** The backend will issue a JWT stored exclusively in `HttpOnly` and `Secure` cookies.
- **Trade-off:** Prevents XSS (Cross-Site Scripting) attacks but requires strict CORS and CSRF configuration.

## 3. Risks and Unknowns (Known Unknowns)

- **Voting Concurrency:** Multiple users voting at the exact same millisecond. The database will mitigate duplicate votes from the _same_ user (via composite PK), but the total vote count (if denormalized in the future) will require transactions (e.g., `SELECT FOR UPDATE`). For the MVP, we will use dynamic aggregated counting.
